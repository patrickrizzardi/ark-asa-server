### What's Solid
This is a clean kill of the 3-way hand-list drift flagged in M2 cumulative review concern #1. The old code carried the AsaApi root-artifact set in THREE places that had to stay in lockstep: the Dockerfile bake `cp` list, the entrypoint rm-list (which had already drifted — it carried a vestigial `AsaApiLoader.pdb` the cp-list didn't), and the entrypoint cp-list. The refactor derives the set ONCE from the image tree (`/opt/asaapi/*`) and drives both rm and cp from it, so the entrypoint now mechanically follows whatever the Dockerfile bakes. The two can no longer drift. This is exactly the structural fix the enumerable-domain principle calls for (derive from source-of-truth, don't hand-list) — applied correctly to a bash artifact-list, which is the harder-to-spot variant. The comment block is durable (describes the current invariant + why derive beats hand-list), names the ArkApi/config.json exclusions and WHY each is excluded, and carries no changelog phrasing. `local` declarations are correct, loop vars are declared, quoting is correct throughout.

### Concerns (non-blocking, but will bite later)
1. **entrypoint.sh:110 — glob-no-match would inject a literal `*` into `root_artifacts` if `/opt/asaapi` were ever empty.** With no `nullglob` set, `for entry in "${src}"/*` over an empty dir iterates the literal string `*`; `basename` returns `*`, it's neither ArkApi nor config.json, so `root_artifacts` gets a member `*`. I traced the blast radius empirically: because the consumer loops QUOTE the expansion (`rm -f "${win64}/${artifact}"` → `rm -f "win64/*"` as a literal filename, not a glob), the rm is a harmless no-op on a nonexistent file named `*` (verified: game files in a populated win64 survived). The cp loop `cp "${src}/*"` would then fail and `set -e` aborts — loud, not silent-wrong. AND `/opt/asaapi` is image-baked (Dockerfile RUN at build, never volume-mounted — the only volume is `ark-game:/home/container/arkserver`), so the bake guarantees ArkApi/ + config.json + 5 DLLs always exist and the glob can never come up empty in production. So this is unreachable today and contained even if reached. NOT a BLOCK. But a one-line `shopt -s nullglob` (or a `[[ -e "${entry}" ]] || continue` guard) at the top of the loop would make the degenerate case produce an empty array instead of a literal `*`, which is the more honest behavior and costs nothing. Flagging as a Concern / optional hardening, not a gate.
2. The ArkShop bake (`cp -r /tmp/arkshop_src/ArkShop/. /opt/asaapi/ArkApi/Plugins/ArkShop/`) targets the ArkApi subtree, not `/opt/asaapi` root, so it cannot add a root artifact the derive would need to pick up. Noted only to document that the derive's completeness was checked against the full bake, not just the explicit root-cp at Dockerfile:47-53.
