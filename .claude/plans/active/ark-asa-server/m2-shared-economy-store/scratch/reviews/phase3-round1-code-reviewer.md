### What's Solid
- **The verify-the-artifact-not-the-exit-code discipline is correctly carried over.** `install_vcredist()` (entrypoint.sh:160-172) does the right thing — it does not trust `proton run`'s exit code as proof of success, it checks the three actual runtime DLLs landed in the prefix system32 and fail-fasts (`exit 1`) with a clear FATAL message naming exactly which DLLs are missing and where it looked. This mirrors the `install_or_update` `$SERVER_EXE` check (lines 38-41) and the comment reference at line 161 is accurate.
- **The skip-gate fix is the genuinely good call here.** Gating on DLL presence (`-f "${msvcp}" && -f "${vcrt}" && -f "${vcrt1}"`) rather than a bare marker file means a `pfx/` reset (prefix nuked + recreated) correctly re-triggers the install. A marker-only gate would have falsely skipped and left AsaApi unable to load. The marker is correctly demoted to a fast-path hint, and the conjunction `marker AND all-three-DLLs` (line 152) means even a present marker can't override missing DLLs. This is exactly the re-run-safety property the scope note asked me to scrutinize, and it holds on warm boot AND after a prefix reset.
- **Dockerfile layer hygiene is clean.** Single `RUN` (mkdir + curl + chown in one layer, line 61-64), `curl -fsSL` (fail-on-HTTP-error, silent, follow-redirects — `-f` is load-bearing here since `aka.ms` is a redirect), chowned to `container:container` so the non-root user can read it at runtime. Baked to a neutral `/opt/vcredist/` path exactly per the established `/opt/steamcmd` + `/opt/asaapi` precedent.
- **`set -u` safety is satisfied.** `STEAM_COMPAT_DATA_PATH` is a hard `ENV` in the Dockerfile (line 74), so the `local pfx_sys32="${STEAM_COMPAT_DATA_PATH}/..."` expansions (lines 144-145) can't trip the unbound-variable abort. Verified.
- **The bash array pattern is correct under `set -euo pipefail`.** `local missing=()` then `missing+=("...")` guarded by `[[ ... ]] || missing+=(...)`, and the `${#missing[@]} -gt 0` test. `printf "  %s\n" "${missing[@]}"` (line 169) is the correct way to dump the array. No empty-array-expansion footgun because the `-gt 0` guard means the `printf` only runs when the array is non-empty. The `|| missing+=(...)` short-circuits are `set -e`-safe (a failing `[[ ]]` on the left of `||` does not abort).
- **ADR 0002 is well-formed and substantive.** Frontmatter (`doc-type: adr`, id, status accepted, date), Context / Decision / Rejected alternatives / Consequences all present. The rejected-alternatives section is real (Dockerfile-install is a hard mechanical constraint not a preference; runtime-download is the anti-pattern this project beats; vendoring weighed with a named fallback). The 3-question table per artifact is the durable substance an ADR should carry. Registered `[locked]` in `.claude/design-sources.md` in the same change. ADR 0001 exists, so the numbering is correct.
- **Comments follow the durable-not-changelog standard.** The `install_vcredist` header is Tier-3 appropriate (external I/O into a volume prefix): states WHY runtime not build-time, the skip-gate rationale, Time/Space, and Side effects. No banned changelog phrases. The one "originally said Dockerfile / was stale" phrasing lives in the **rule doc + ADR** (build-time-vs-runtime.md note, ADR Consequences), which is the correct home for a design-correction narrative — that's documentation explaining a corrected decision, not a code comment narrating a diff. Acceptable.

### Concerns (non-blocking, but will bite later)

1. **`proton run` non-zero exit aborts the script BEFORE the verify block runs (entrypoint.sh:158).** Under `set -e`, if `proton run /opt/vcredist/VC_redist.x64.exe` exits non-zero, the script dies at line 158 — the carefully-written DLL-verify fail-fast at lines 162-172 never executes. In practice this is mostly fine (a hard `proton run` failure is still a fatal abort, which is the desired outcome), and the launch `proton run` at line 220 sidesteps `set -e` only because it's backgrounded with `&`. But the comment at line 160-161 ("The installer's exit code cannot be trusted as proof") implies the code deliberately proceeds past a questionable exit code to let the DLL check be the arbiter — and `set -e` silently prevents that on a *non-zero* exit. The DLL check only actually arbitrates the **exit-zero-but-DLLs-missing** case (the real footgun the comment is about), which it does correctly. If you want the exit-nonzero case to also reach the richer FATAL message instead of bash's bare abort, append `|| true` to line 158 and let the DLL gate be the sole arbiter. Not a correctness bug — the script still fails closed either way — but the code's intent and `set -e`'s behavior are slightly out of sync. Worth a one-line `|| true` to make the verify block the single source of truth it's written to be.

2. **VC++ redist version label may be imprecise — `aka.ms/vs/16` (Dockerfile:62) vs the "VC++ 2019 (14.2x)" claim.** Microsoft has historically re-pointed the `aka.ms/vs/16/release/vc_redist.x64.exe` URL to serve the merged 2015–2022 runtime (14.4x at current), not a frozen 2019 14.2x build. The plan/comments label it "VC++ 2019". This is cosmetic and self-correcting: AsaApi needs `msvcp140.dll` / `vcruntime140.dll` / `vcruntime140_1.dll`, all of which the merged runtime provides, and the DLL-presence gate verifies them by name regardless of which marketing-year build installed them. So the install can't silently regress. But two follow-ups: (a) the URL is **not version-pinned** — `aka.ms/vs/16` always serves whatever Microsoft currently hosts there, which contradicts the ADR's "version-pinned / image is the version source-of-truth" framing (the `ARG`-pin discipline used for AsaApi is absent here; the redist is pinned only to "whatever 14.x is live"). If reproducibility of the exact redist build matters, pin a versioned download URL or vendor the binary. (b) Correct the "2019 / 14.2x" label in the comment + ADR to "VC++ 2015–2022 redistributable (14.x)" so the next maintainer isn't misled. Cross-flag to plan-adherence-verifier if the plan text asserts a specific 14.2x build as an acceptance criterion.

3. **`deploy_plugins` (line 96, pre-existing, not Phase 3) already `cp`s `msvcp140.dll` into Win64 — note the two are different DLL locations, not a conflict.** Flagging only so the next reader doesn't think there's a double-install collision: `deploy_plugins` copies `msvcp140.dll` next to `AsaApiLoader.exe` in the game's `Win64/` (an app-local DLL beside the loader), whereas `install_vcredist` registers the runtime into the **Wine prefix** `drive_c/windows/system32`. Different paths, different mechanisms, both legitimately needed. No action — just confirming I traced it and it's not a duplicate.
