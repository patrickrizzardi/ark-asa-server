### Concerns (non-blocking, but will bite later)

1. **[entrypoint.sh:22-23, 316] Dead fallback tier + misleading FATAL guidance for DB host/port.** The fallback chain reads `${MARIADB_HOST:-mariadb}` and `${MARIADB_PORT:-3306}`, but `MARIADB_HOST`/`MARIADB_PORT` are defined **nowhere** — not in `docker-compose.yml`, not in either `.env.example`, and not in the game-service env passthrough block (only `MARIADB_DATABASE`/`MARIADB_USER`/`MARIADB_PASSWORD` are plumbed through, plus the `ARKSHOP_DB_*` set). So the `MARIADB_HOST`/`MARIADB_PORT` middle tier is dead — it always falls through to the literal `mariadb`/`3306`. The literal default IS correct (service name + standard port), so there's **no wrong-output bug** — this is why it's a Concern, not a BLOCK. But: (a) the intermediate tier is dead code per `no-duct-tape.md` #4 (a fallback reading a var that's never set); and (b) the FATAL message at line 316 actively misleads — it tells the operator their creds "default to MARIADB_HOST/USER/PASSWORD/DATABASE," but an operator who sets `MARIADB_HOST=x` in their `.env` following that message gets it **silently ignored** (compose never passes `MARIADB_HOST` to the game service; only `ARKSHOP_DB_HOST` would take effect). Compare the name/user/pass tiers, which correctly chain to the real `MARIADB_DATABASE`/`MARIADB_USER`/`MARIADB_PASSWORD` vars — those are live. Fix: either drop the dead `MARIADB_HOST`/`MARIADB_PORT` tier (collapse to `${ARKSHOP_DB_HOST:-mariadb}`) and correct the line-316 message to name only the vars that actually exist, OR add `MARIADB_HOST`/`MARIADB_PORT` to the compose passthrough + `.env.example` if a real host/port override was intended. The asymmetry (name/user/pass chain to real vars; host/port chain to phantoms) is the smell.

2. **[entrypoint.sh:296, 334, 357] Comment mischaracterizes the password-passing mechanism; undocumented argv exposure.** The header comment says the password reaches jq "via env substitution, and jq does not log its argument values." It's actually passed as a command-line **argument** (`--arg pass "${ARKSHOP_DB_PASS}"`), which lands in jq's argv — visible in `/proc/<pid>/cmdline` and `ps aux` for the duration of the jq exec. In this single-tenant container (one user, no untrusted co-tenants in the namespace) that's a negligible vector, so not a BLOCK — but the "env substitution" wording is wrong and the argv exposure is undocumented. Fix: correct the comment to "passed as a jq `--arg` command-line argument (transiently visible in argv during the jq exec; acceptable in this single-user container)," or pass via `jq --rawfile`/env+`env.PASS` if you want to keep it out of argv entirely.

3. **[entrypoint.sh:333-341] jq silently creates a phantom `.Mysql` block if the real config schema differs (casing/nesting).** `jq '.Mysql.UseMysql = true | ...'` will happily create a brand-new `.Mysql` object if the seeded ArkShop config nests its MySQL settings differently (e.g. `.MySQL`, or under a parent key). ArkShop would then ignore the injected phantom and run file-backed with `UseMysql=false` — a **silent** DB-less shop. The jq filter casing matches plan research finding #2's cited schema (`ark-server-api.com/resources/asa-arkshop.34`: `Mysql`/`UseMysql`/`MysqlHost`/...), and the default config isn't committed (it ships inside the baked ArkShop zip), so I can't byte-verify the schema in this env — static-evidence ceiling. This is exactly what dell-boot AC1 (no `Singleton not found`, no DB error) + AC3 (row persists) will catch. Flagging so the dell-boot verifier knows a clean `ArkApi.log` is load-bearing here: if the schema casing is off, the symptom is a silent file-backed shop, not a crash. No code change needed if AC1/AC3 pass on dell.

### Section 4 — Documented Deviation Adversarial Validation

**SCOPE #1 (Dockerfile +jq apt layer):** Read cold — `jq` added to the existing `unzip` apt RUN. Adversarial input: does jq actually exist in the entrypoint's runtime PATH (build-as-root, run-as-container)? `apt install` to `/usr/bin/jq` is world-executable; container user can invoke it. Correct placement per build-vs-runtime (immutable build dep → Dockerfile). VALIDATED — no overshoot.

**SCOPE #2 (.gitignore plugins-config/** except .gitkeep):** Read cold. Adversarial input: a SECOND plugin config (`Permissions/config.json`) also gets a password injected — is it covered, or does the glob only catch ArkShop? `git check-ignore` confirms BOTH `ArkShop/config.json` and `Permissions/config.json` are ignored by `plugins-config/**`; `.gitkeep` tracked. The negation `!plugins-config/.gitkeep` doesn't accidentally un-ignore the password files (tested). VALIDATED — the wider rule (`/**`) is correct, not an overshoot.

**APPROACH #2 (jq --arg for creds, not sed):** Read cold. Adversarial input: a password containing jq-filter metacharacters AND shell metacharacters, e.g. `p@ss"$(rm -rf /)'|.x`. `--arg pass "$VALUE"` binds it as a literal string variable inside the filter — never re-parsed as a filter expression, never re-evaluated by the shell (already a bash variable). The `tonumber` coercion on port is the only place a non-string is built, and port comes from a numeric-defaulted var. VALIDATED — jq --arg genuinely defends the injection surface sed would have exposed.

**APPROACH #3 / the flagged "deploy_plugins ↔ setup_plugin_configs warm-boot no-op" claim:** This is the one the coordinator asked me to validate adversarially. Executor claims (notes.md): "deploy_plugins() stash-restore is now a warm-boot no-op for symlinked plugins (stashes from host-bind, restores into real dir, then setup_plugin_configs() rm+symlinks over it). Host-bind file is never corrupted."

I did NOT test the case the executor named (clean warm boot). Adversarial input constructed: **warm boot where the host-bind config carries the injected live password, and deploy_plugins runs FIRST (main line 399) before setup_plugin_configs (line 403).** Traced step-by-step against verified `rm -rf`/symlink/glob semantics (I ran the symlink behavior in a scratch dir to confirm):

1. Prior boot left `${win64}/ArkApi/Plugins/ArkShop` as a **symlink** → `/home/container/plugins-config/ArkShop` (host bind), whose `config.json` holds the injected password.
2. deploy_plugins stash loop globs `ArkApi/Plugins/*/config.json` — confirmed the glob resolves *through* the symlink and `[[ -f ]]` is true, so it stashes the **password-bearing host config** into a `mktemp -d` (mode 700) tmpdir. Password transits a 700 dir — acceptable.
3. `rm -rf "${win64}/ArkApi"` — confirmed in scratch test: removing a parent dir that contains a symlink removes the **link**, not the link's target. The host bind at `/home/container/plugins-config` is OUTSIDE `ArkApi/` → **host data survives.** (Note: the code uses `rm -rf "${win64}/ArkApi"` with no trailing slash on a real dir — safe. A trailing slash on a *symlink* would delete through it, but ArkApi itself is always a real dir here, and the per-plugin symlinks are removed as links when their parent ArkApi is wiped. No delete-through-link path exists.)
4. `cp -r src/ArkApi` lays down fresh real dirs with image-default configs.
5. Restore loop copies the stashed (password-bearing) config back into the fresh real `ArkShop/` dir.
6. setup_plugin_configs: host `config.json` already exists → seed-if-absent skips (correct, no clobber of operator edits); `rm -rf plugin_dir` (now a real dir) → `ln -sfn host_dir`.
7. inject_plugin_db_config re-injects (idempotent — same creds, same output).

**Conclusion: the "no-op / host-file-never-corrupted" claim holds.** The host bind is the durable source of truth; deploy_plugins round-trips its content through a 700 tmpdir and the symlink is rebuilt every boot. The redundancy (stashing a config that's about to be re-symlinked and re-injected anyway) is genuinely harmless, not a bug. Adversarial inputs attempted: (a) password-bearing host config warm-boot round-trip, (b) delete-through-symlink via trailing-slash rm, (c) seed-if-absent clobber of operator edits on warm boot. None broke the fix. Deviation VALIDATED.
