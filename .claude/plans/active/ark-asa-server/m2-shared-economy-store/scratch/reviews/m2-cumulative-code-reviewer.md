### What's Solid
- **Entrypoint ordering is correct and defensible.** `install_or_update → deploy_plugins → install_vcredist → [if modded: ensure_modded_pdb → setup_plugin_configs → inject_plugin_db_config → MODS-append] → launch`. The unconditional steps (deploy/vcredist) run before the modded-only block, and every modded-only step depends on the prior one's output. Traced fresh-volume first boot and warm boot — no ordering hole.
- **The warm-boot config-stash interaction is sound** (the thing most likely to be broken in a clean-replace + symlink design). On boot 2, `ArkShop/config.json` is already a symlink to the host bind. `deploy_plugins()` line 95 globs it, `[[ -f ]]` follows the link and stashes the host CONTENT; `rm -rf ArkApi/` removes only the link (host target survives); fresh `cp -r` lays down a real file; stash-restore overwrites with the operator's content; then `setup_plugin_configs()` re-symlinks and `inject_plugin_db_config()` re-injects idempotently. No DLL loss, no orphaned host config, no double-seed. Clean.
- **Idempotency holds across the whole sequence.** VC++ skip-gate keyed on actual prefix DLLs (not a bare marker) survives a `pfx/` reset — genuinely better than a marker-only gate. `ensure_modded_pdb()` size-floor (>1 MiB) rejects a truncated pdb that a bare `-f` would pass — closes the silent-zero-plugin failure. `_inject_mysql_block` resolves the symlink via `readlink -f` before `mv` so it never replaces the host-bound link with a regular file. These are the three subtle idempotency traps in this kind of entrypoint and all three are handled.
- **Secret handling is correct.** Creds flow env → `jq --arg` (special-char-safe, never a jq filter), password never echoed (inject logs host/db/user only, line 380-381), `.env*` gitignored with `.example` placeholders, `plugins-config/**` gitignored except `.gitkeep`, compose uses `${VAR:?err}` fail-loud on the two real secrets. `git grep` for the password finds nothing (AC2 evidence). The transient `/proc/<pid>/cmdline` exposure of the password during the jq exec is correctly judged acceptable for a single-user game container and documented as such.
- **Fail-fast discipline is consistent.** Missing/empty creds → FATAL; non-numeric port → FATAL before it reaches jq's `tonumber`; missing ArkShop config → FATAL; jq write failure → explicit `exit 1` (the load-bearing `|| { … }` in the non-final `&&` position where `set -e` won't fire). "Trust the artifact, not the exit code" applied uniformly (steamcmd marker, VC++ DLL check, pdb size-floor).
- **Kill-switch is real, not a flag graveyard.** `ENABLE_ASAAPI=0` restores byte-for-byte M1 vanilla launch (verified AC3 on dell), documented in both `.env.*.example`, permanent operational toggle (not a rollout flag with a cleanup deadline). Defaults consistent: entrypoint `:=1`, compose `:-1`.
- **Docs are deliverables, not afterthoughts.** ADR 0001 (MariaDB engine, MySQL-8.0.28 rejection, rejected alternatives), ADR 0002 (bake-to-/opt + runtime-deploy pattern with the 3-question test), `build-time-vs-runtime.md` amended so the rule no longer contradicts the volume-backed-prefix code, design-sources registry bootstrapped with all three `[locked]`. No rule/code contradiction remains.

### Concerns (non-blocking, but will bite later)

1. **[entrypoint.sh:105-122 + Dockerfile:47-53] THREE hand-maintained enumerations of the same closed domain (the AsaApi root-artifact set) — make the deploy derive from the image tree so the set lives in ONE place. `[enumerable-domain]`**
   The "AsaApi root-level artifacts" set is hand-listed in three independent places:
   - **Dockerfile:47-53** (cp into `/opt/asaapi/`): `AsaApiLoader.exe msvcp140.dll msdia140.dll libcrypto-3-x64.dll libssl-3-x64.dll config.json`
   - **entrypoint.sh:105-111** (rm-list): `ArkApi/ AsaApiLoader.exe AsaApiLoader.pdb msdia140.dll libcrypto-3-x64.dll libssl-3-x64.dll msvcp140.dll`
   - **entrypoint.sh:117-122** (cp-list): `AsaApiLoader.exe msdia140.dll libcrypto-3-x64.dll libssl-3-x64.dll msvcp140.dll`

   These are already NOT identical (rm-list carries `AsaApiLoader.pdb` which the build strips and the cp-list omits — benign today, but it proves the lists drift independently). The failure mode is a future AsaApi version bump that adds a new root DLL (e.g. `libffi.dll`) or drops one: the maintainer must edit all three lists in lockstep. Miss the **rm-list** → a dropped DLL lingers stale on the volume across a version bump (the exact "clean-replace, no stale files" property AC4 of Phase 2 was supposed to guarantee — silently violated). Miss the **cp-list** → a newly-added DLL is baked in `/opt` but never deployed → AsaApi fails to load with a confusing missing-dependency error. Miss the **Dockerfile** → the bake itself drops it. This is `no-duct-tape.md` #3 (hardcoded list assumes the world stays at N) and the enumerable-domain hand-list anti-pattern: report the structural fix, not "add the next DLL."

   **WHY this is a Concern and not a BLOCK:** it is not silent-wrong-output *today*. AsaApi is pinned at v1.21, all three lists are correct for v1.21, and the full deploy→launch→connect path was verified live on dell. The bug requires BOTH a future version bump that changes the root set AND a maintainer updating only some of the lists. That is latent maintainability drift, not a present defect — Concern per the rubric (silent wrong-output *now* would be a BLOCK; a future-conditional one is not).

   **WHY it must be surfaced loudly anyway:** this exact concern was flagged at Phase 2 round 1 ("dual-static-list maintenance coupling"), deferred to "Phase 5," and Phase 5 never touched `deploy_plugins()` — it dealt with ArkShop config injection. The deferral fell through the cracks and reached end-of-plan unaddressed. Per `no-duct-tape.md`, a deferral with no written anchor that survives to ship is exactly the pattern the rule prohibits; it should be either fixed now or recorded as an explicit, anchored follow-up (capability-ledger row owned by a real later milestone) — not silently carried.

   **FIX (structural, not whack-a-mole):** derive the deployed set from the image tree so the enumeration exists once. e.g. build the set from `/opt/asaapi`'s top-level entries minus `config.json` (which is seed-if-absent and must be excluded from the wipe), and drive BOTH the `rm` and the `cp` from that one derived list. Concretely: `for entry in /opt/asaapi/*; do name=$(basename "$entry"); [[ "$name" == config.json ]] && continue; rm -rf "${win64}/${name}"; cp -r "$entry" "${win64}/"; done`. That collapses the two entrypoint lists into one image-tree-driven loop; the Dockerfile bake remains the single upstream source the loop reads from, eliminating the entrypoint-side drift entirely (Dockerfile + one derived loop = the bake list IS the deploy list). `config.json` stays special-cased exactly as today (line 140 seed-if-absent). Tag for the orchestrator's same-class backstop: `[enumerable-domain]`.

2. **[entrypoint.sh:386] Permissions Mysql-block injection is a soft `has("Mysql")` gate — confirm Permissions V1.1 actually persists role grants to MariaDB, or this is dead code.** `inject_plugin_db_config()` injects the Mysql block into `Permissions/config.json` only `if … jq -e 'has("Mysql")'` succeeds. If the bundled Permissions V1.1 default config has no `Mysql` key, this branch never fires — Permissions would run file-backed, not DB-backed, silently. The live dell run shows ArkShop tables created in MariaDB (AC1) but the evidence does not separately confirm Permissions writes to the DB. Not a code-quality defect (the soft gate is the correct defensive shape — it can't break a config that has no Mysql block), but worth a one-line runtime confirmation that Permissions role grants actually land in MariaDB, else the Permissions-DB wiring is aspirational. Cross-ref acceptance-verifier.

3. **[entrypoint.sh:502-511, pre-existing M1] stop() runs SaveWorld then immediately `kill server_pid` with no completion wait, and kills only the proton wrapper PID.** On `SAVE_ON_STOP=1` (prod default) the RCON SaveWorld is fired and the very next line kills the server — there's no wait for SaveWorld to finish flushing, so a large world could be killed mid-save. Also `kill server_pid` targets the backgrounded `proton run` wrapper; the Wine/`AsaApiLoader.exe` process tree under it may not receive the signal. **This is pre-existing M1 behavior** (verified against `873509a:entrypoint.sh` — identical `kill server_pid` shape on a backgrounded `proton run`), NOT an M2 regression, so it is out of scope for this milestone's diff. Noting it as a future hardening candidate (a `stop_grace_period`-bounded SaveWorld-completion wait), not an M2 finding.

4. **[entrypoint.sh:336, pre-existing-style] `mktemp` lands in `/tmp` (container writable layer) while the inject `dest` is on the `./plugins-config` bind mount → `mv` is cross-device (copy+unlink, non-atomic).** For a single-user game container booting serially, before the loader launches and reads the config, there is no concurrent reader, so the non-atomicity is inconsequential. Carried verbatim from Phase-5 round-3 review; noted so a future maintainer who adds concurrency knows the write isn't atomic. Not a fix.
